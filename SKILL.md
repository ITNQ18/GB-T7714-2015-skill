## 必须执行的启动规则

只要本技能被触发，必须读取并执行本技能规则。不要因为用户没有重复说明格式要求，就跳过本技能中的默认规则。同一次对话中重复触发本技能时，首次触发必须完整读取 `SKILL.md`；后续每隔两次触发再完整读取一次，即第 1、4、7…… 次触发完整读取，第 2、3、5、6…… 次触发按任务需要读取相关章节。读取是为了理解和遵守规则，不得把 `SKILL.md` 全文通过 Python、PowerShell 或其他命令打印到对话上下文；只在需要向用户解释或排查时摘要说明相关规则。

默认执行以下规则：

- 读取本技能规则、项目状态、参考文献概览、JSON 清单和 Markdown 文档时，必须显式使用 UTF-8 编码，避免 Windows 终端默认编码导致中文规则乱码。读取技能规则可在内部完成，不等于把全文输出到上下文；若使用命令检查文件内容，避免无必要地整篇打印。PowerShell 读取示例：`Get-Content -Raw -Encoding UTF8 <文件路径>`；Python 读取示例：`Path(path).read_text(encoding="utf-8")`。
- 运行本技能脚本前必须完成环境准备：先检查项目根目录是否已有 `.venv/`、`venv/`、`env/` 或 `.env/`；若存在，使用其中的 Python 运行脚本并先重试失败命令；若不存在，则在项目根目录创建 `.venv`，升级 `pip`，安装本技能依赖 `python-docx`、`lxml`，再继续执行。
- 文献必须真实、可核验。
- 使用顺序编码制。
- 正文引用为右上角上标。
- 正文引用默认使用 Times New Roman 小四字体（12 pt）；若用户、学校模板或目标文档明确指定其他字体，则以明确要求为准，并在最终回复中说明。
- 正文引用必须优先使用 Word“编号项”交叉引用，引用内容为“段落编号”。只有当 Word 编号项交叉引用生成失败时，才允许降级为书签 REF 方案，并必须在最终回复中说明 fallback。
- 参考文献表必须使用真实 Word 编号项，不得手打 `[1]`、`[2]` 冒充编号。
- 参考文献条目默认使用悬挂缩进两个字符。若目标文档已有格式与本技能默认格式冲突，必须先向用户说明具体差异（例如字体、字号、行距、对齐、首行缩进、左缩进、悬挂缩进），并询问保留已有格式还是改为默认格式。
- 正文引用编号必须与参考文献编号一致。
- 每条参考文献默认都必须被正文引用，除非用户明确允许未引用文献。
- 默认禁止使用第 6 条电子文献格式 `[EB/OL]`。若用户或学校模板明确要求禁用电子文献，则所有最终写入条目必须属于期刊、专著、会议论文、论文集中析出文献、学位论文或专利格式；找不到正式、可查询、可核验的出版记录时必须向用户说明，不能把电子来源伪装成正式文献。
- 删除文献时，必须删除对应正文引用或删除失效编号，并重新按正文首次出现顺序编号。
- 可以拆分的长串引用应按语义拆分；单个引用点默认最多 3 条文献。
- 涉及收集、添加、补充或写入参考文献时，必须确认 `所需数量`。若项目状态、用户指令和目标文档均未明确需要多少条参考文献，一定要先向用户提问，不得自行决定数量后继续执行。
- 不进行渲染和页面图像检查；机器审计仅以 OOXML、字段、编号、样式和文本抽取为依据，最后直接建议用户在 Word/WPS 中全选更新域并人工复查。
- 技能触发时，若 `参考文献/` 文件夹不存在则创建；若 `参考文献/任务状态.json` 不存在则写入默认值；若 `参考文献/参考文献概览.md` 不存在则创建带表头的文件，若 DOCX 已有文献则抽取填充。后续添加或删除文献时，同步更新这两个文件。
- 交付纪律参考 Documents 技能：允许生成内部 QA/处理中间文件，但它们不是交付件。用户请求 DOCX 时，最终只交付或链接最终 DOCX，并说明必要状态/报告；不要把临时 JSON、占位 DOCX、渲染/审计中间件、Word/OOXML 临时副本作为最终成果交给用户，除非用户明确要求查看。
- 内部 QA/处理中间文件必须使用清晰的临时位置或临时命名，避免与最终 DOCX 混淆。可以短暂生成带 `[[CITE:key]]` 的占位 DOCX、临时 `references*.json`、Word 编号项基底或 OOXML 副本来支撑脚本执行和审计，但默认在最终 DOCX 生成并完成审计后清理；保留最终 DOCX、原始草稿、`参考文献/任务状态.json`、`参考文献/参考文献概览.md`、审计报告和拆分报告。若用户明确要求保留中间文件用于调试或复核，则可以保留并说明其非交付性质。

如果用户的即时要求与本技能默认规则冲突，优先执行用户的明确要求，并在最终回复中说明偏离了哪条默认规范。

## 任意入口续作规则

用户可能从任意阶段开始提问或要求修改。先识别当前状态，再继续执行，不要假设必须从"收集文献"开始。

进入任何任务时，先做轻量状态识别：

1. 确认目标文件：优先使用用户指定的 DOCX；若未指定，使用当前会话最近处理的 DOCX；仍不明确时再询问。
2. 读取项目状态：若存在 `参考文献/任务状态.json` 和 `参考文献/参考文献概览.md`，先读取；若不存在，创建 `参考文献/` 文件夹，写入默认 `任务状态.json` 和带表头的 `参考文献概览.md`，若 DOCX 已有文献则抽取填充，不阻塞任务。
3. 确认所需数量：当入口属于收集、添加、补充、写入或生成参考文献，且 `任务状态.json` 的 `所需数量` 为空、缺失或无法从用户明确指令/学校模板/目标文档要求推出时，必须先向用户提问“需要多少条参考文献？”；用户回答前不得开始检索、筛选或写入文献。若用户明确表示“由你决定数量”，再按正文引用点需要确定数量，并把原因写入状态文件。
4. 抽取 DOCX 结构：读取正文引用、参考文献标题、参考文献编号项、交叉引用字段、编号定义和已有文献条目。
5. 判断用户入口属于哪类：收集、插入、删除、改格式、拆引用、补引用、审计、继续上次任务、只回答问题。
6. 只询问完成任务必须的信息；但参考文献数量是收集、添加、补充、写入或生成参考文献任务的必须信息，不能因为可自行估算就跳过提问。

## 入口处理矩阵

- 用户要求"添加/补充参考文献"：先识别正文主题和可引用位置，再检索核验真实文献，更新概览和状态文件，插入正文引用和参考文献编号项。
- 用户要求"修改/删除参考文献"：先定位受影响条目和所有正文引用，删除或替换后重新按正文首次出现顺序编号。
- 用户要求"检查是否都引用"：抽取所有参考文献编号项和正文交叉引用，报告未引用、越界引用、重复异常；若用户要求修复，再补引用或删除未引用项。
- 用户要求"拆开引用/太密集"：先读取相关段落上下文和对应文献条目，把引用移动到其支撑的短语、名词、模型名、方法名、工具名或明确论断后；每个引用点最多 3 条，能拆则拆，并尽量避免多个引用覆盖同一大段句意。
- 用户要求"设置交叉引用"：先把正文普通 `[n]` 或旧字段转换为 Word“编号项/段落编号”交叉引用，保持上标和指定字体；如果编号项生成失败，再使用书签 REF fallback 并说明原因。
- 用户要求"改字体/字号/上标"：只改正文引用运行的格式，不改文献内容和编号关系，除非发现字段损坏必须同步修复。
- 用户要求"审计"：只做机器审计和报告，不改文件，除非用户明确要求修复。
- 用户只给局部截图或段落：优先局部处理，同时检查该局部修改是否破坏全局编号和引用覆盖。
- 用户从中途继续：读取状态文件、概览文件和当前 DOCX，不重新询问已知信息，不从头重做。

## 最高优先级规则

文献真实性优先于数量。不得编造作者、题名、日期、期刊、会议、卷期、页码、DOI、URL、出版社、出版地或任何其他著录字段。

如果用户要求 50 条但只能核验 43 条，只写入 43 条，并说明其余未写入原因。不得为了凑数量加入无关或无法核验的文献。

默认不使用第 6 条电子文献格式 `[EB/OL]`。电子资源、网页、官方文档、CTAN、GitHub、arXiv、OpenReview、Hugging Face paper 页面等只能作为检索线索或核验入口；只有能核验到正式、可查询的出版记录时，才可按对应类型写入最终参考文献。

不得把 arXiv、网页、CTAN、官方文档、项目主页、预印本页面或其他电子页面伪装为期刊、会议论文或论文集中析出文献。最终写入的文献必须同时满足两点：格式符合用户给出的 7 类模板之一，来源具有可查询、可核验的正式出版记录。

若某个引用点只能找到电子资源或预印本，必须向用户说明“无法按当前规则写入”，并列出可选处理方式：跳过该文献、替换为支撑同一论述的正式出版文献，或等待用户明确授权改用第 6 条电子文献格式。

注意区分“检索可见”和“最终写入”：检索阶段不得隐藏 arXiv、OpenReview、网页、项目主页、官方文档等电子线索，应保留给模型阅读、比较和判断；默认限制只作用于最终写入。若用户或学校模板明确允许 `[EB/OL]`，可在 MCP 中设置 `allow_electronic=true` 或脚本中使用 `--allow-electronic`，但写入前仍必须由模型确认权威性、相关性、发表/更新日期、引用日期和访问路径。

不得使用普通文本、空格、截图、文本框、手打编号等方式模拟真实编号或交叉引用。

## 文献收集与核验

1. 阅读正文，识别需要引用的位置：
   - 研究背景、相关工作、理论基础。
   - 算法、模型、框架、工具、协议、标准。
   - 数据集、实验方法、评价指标。
   - 软件库、官方文档、电子资源对应的正式论文、专著、会议论文或专利。
   - 与项目核心主题高度相关的段落。
2. 为每个引用点寻找文献：
   - 每个引用点必须支撑具体短语、子句、方法、模型、工具或论断。
   - 优先经典、高影响、原始提出方法的文献。
   - 近 3-5 年文献优先使用正式发表版本；无正式版本时不得默认使用预印本，必须向用户说明并等待确认或替换为正式相关文献。
   - 不为凑数量添加无关文献。
3. 核验来源优先级：
   - DOI / Crossref。
   - 出版社、期刊、会议官网。
   - IEEE、ACM、Springer、Nature、Science、Elsevier 等出版方页面。
   - ACL Anthology、CVF Open Access、PMLR 等可查询的正式出版记录。
   - 中文文献优先 PubScholar、CNKI 公开记录、万方、维普、国家哲学社会科学文献中心、学校或机构库。
   - 官方标准、官方文档、CTAN、项目官网仅作线索；默认不得作为最终 `[EB/OL]` 条目。
4. 版本选择：
   - 同一文献同时存在 arXiv、会议版、期刊版时，优先正式发表版本。
   - 不混用同一文献不同版本的元数据。

网络检索、来源分层、7 类模板路由、代理设置、拒收条件和部分成功报告见 `references/literature-search.md`。凡执行收集、替换、补充、核验或写入参考文献任务，必须先读取该文件；不能把 arXiv、网页、项目主页或官方文档直接当成正式文献。模型本身没有联网能力时，也必须通过脚本、MCP 或其他外部检索工具搜索；本技能默认代理为 `http://127.0.0.1:7897`。

## MCP 与外部检索

当模型没有原生联网能力，或需要把检索能力暴露成结构化工具时，使用 `mcp-server/gb_reference_server.py`。MCP 只负责搜索、候选分类和写入前核验，不负责修改 DOCX；DOCX 写入、交叉引用和审计仍由 `scripts/应用参考文献.py`、`scripts/审计参考文献.py` 等脚本完成。

MCP 工具与安装方式见 `references/mcp-search.md`。默认 MCP/脚本检索代理为 `http://127.0.0.1:7897`，可用 `GBT7714_PROXY` 或工具参数覆盖。若 MCP 依赖不可用，退回 `scripts/检索正式文献.py`，但不得跳过正式来源核验。

无联网模型（通过 MCP/脚本代理检索）推荐调用顺序：

1. `search_and_verify_citation_points`：给定一组引用点，自动生成检索词并批量检索 + 核验，返回“可写入候选 + 失败清单”。
2. `build_manifest_draft_from_points`：把上一步输出转成 manifest 草稿，保留核验来源并标注待补字段。
3. `search_formal_references` / `filter_writable_candidates` / `classify_gbt7714_candidate`：用于单条复核、补检索或人工精修。
4. 对可写入候选补齐 GB/T 7714 字段，更新 `参考文献/参考文献概览.md`，再进入 DOCX 写入脚本。

当没有现成参考文献表时，不要求用户手写引用点 JSON。模型应先阅读正文，自动抽取需要引用的短语、方法、工具、模型或明确论断，生成内部引用点；若模型确信该论断对应某篇经典/原始文献，可填 `title_hint`，否则只填核心关键词和 `reference_intent`。脚本返回的 `point_relevance.missing_or_risky` 必须为空，候选才可进入最终 manifest。

## 编号与插入规则

使用顺序编码制：

- 参考文献按正文首次引用顺序排列。
- 正文引用编号必须与参考文献编号一致。
- 同一文献在不同位置被多次引用时，编号保持不变，所有引用处均使用相同编号。
- 新增、删除、移动引用后，必须重新计算正文首次出现顺序。
- 每条参考文献必须被正文引用，除非用户明确允许未引用项。
- 每个引用点默认最多 3 条文献；可拆分时尽量拆成更具体、更短的引用点。
- 每个引用点必须支撑具体短语、名词、方法、模型、工具或明确论断。引用位置首选放在被支撑的短语、名词、方法名、模型名或工具名之后；只有无法自然落到短语级位置时，才放在子句或完整论断之后。
- 避免多个引用点覆盖同一大段文本或同一句中的相同语义范围；总体目标是让引用支撑范围尽量短、清楚、不重叠。
- 如果多篇文献共同支撑不可分割的同一句论断，可保留组合引用，但不超过 3 条。

删除文献时：

- 若一个引用点只引用被删文献，删除该引用点。
- 若一个引用点包含多个编号，只删除失效编号。
- 删除后重排参考文献编号和所有正文交叉引用。
- 不得留下空引用、越界引用或无对应参考文献的编号。

拆分拥挤引用时：

- 先读取段落语义和对应文献条目。
- 把引用移动到被支撑的模型名、工具名、方法名、名词或短语后；只有短语级落点不自然时，才放在子句或完整论断后。
- 多个引用支撑不同内容时，优先拆到不同短语或名词后，避免都堆在句末或同一子句末尾。
- 不为了拆引用改写事实。
- 拆分后检查原长串是否消失、每组是否不超过 3 条、编号是否仍匹配、各引用支撑范围是否尽量不重叠。

## DOCX 交叉引用实现规则

### 默认结构

参考文献表必须使用真实 Word 编号项。正文引用必须优先使用 Word“编号项/段落编号”交叉引用；编号项生成失败时允许书签 REF fallback。

目标结构必须等价于 Word 中“交叉引用 -> 编号项 -> 段落编号”：

- 引用类型：编号项。
- 引用内容：段落编号。
- 插入为超链接：可选，默认可保留。
- 显示格式：普通上标 `[` + 编号项交叉引用字段显示的数字 + 普通上标 `]`。
- fallback：只有编号项生成失败时，才可使用 `REF bookmark` 书签方案；不得用普通文本或超链接锚点替代交叉引用。

组合引用显示为：

```text
[1,2,3]
```

其中每个数字都应是独立的编号项交叉引用字段，逗号和方括号为普通上标文本。

### 编号项优先与书签 fallback

优先使用 Word 自动化插入“编号项/段落编号”交叉引用。旧文档已经存在书签时，可以只读识别并报告；修复或新建时先尝试迁移为编号项交叉引用。

使用 Word COM 生成编号项交叉引用前，参考文献编号项后必须保留一个非编号的普通尾随段落，并确保最后一条参考文献不是直接贴着文档末尾或 `sectPr`。否则 Word 可能能列出最后一个编号项，却在 `InsertCrossReference` 插入最后一项时抛出“命令失败”，导致误判为编号项交叉引用不可用。

如果当前脚本或工具无法生成编号项交叉引用，或 Word 自动化失败，允许使用书签 REF 方案作为 fallback。fallback 必须满足：参考文献条目仍是真实 Word 编号项，正文引用仍是 Word 字段，最终回复说明“已降级为书签 REF fallback”。

### 字段更新

修改字段后，应设置 Word 打开时更新字段的文档设置。无法触发 Word 实际刷新时，必须保证字段指令、显示文本和参考文献编号当前一致，并提醒用户在 Word/WPS 中全选后更新域。

## 字体与格式规则

正文引用：

- 默认上标。
- 默认 Times New Roman 小四字体（12 pt）。
- 引用方括号、逗号和编号字段显示文本必须使用同一字体、字号和上标设置。
- 若用户、学校模板或目标文档明确指定其他字体，按明确要求执行；审计时将其标记为"用户/模板指定"，不要误报为错误。

参考文献标题：

- 标题为"参考文献"。
- 居中。
- 黑体三号（16 pt）。
- 段前 24 pt，段后 12 pt。

参考文献条目：

- 中文宋体 10.5 pt（五号）。
- 英文 Times New Roman 10.5 pt（五号）。
- 1.25 倍行距。
- 无首行缩进、无左缩进，悬挂缩进两个字符，段落双端对齐。
- 使用真实 Word 编号项显示 `[1]`、`[2]` 等编号。
- 每条以英文句点 `.` 结束。
- 半角括号、逗号、冒号和句点；括号、逗号、冒号后空一格；条目末尾句点后不加空格。

## 著录格式

最终参考文献必须严格符合以下 7 类模板之一，但默认禁止使用第 6 类电子文献格式。写入、审计和修复时，优先使用第 1、2、3、4、5、7 类；只有用户明确要求或学校模板明确允许时，才可临时使用第 6 类，并必须在最终回复中说明偏离了本技能默认规则。

1. 期刊格式

```text
[序号] 作者. 题名[J]. 刊名, 年, 卷(期): 起-止页码.
```

2. 专著格式

```text
[序号] 著者. 书名[M]. 出版地: 出版者, 出版年: 起-止页码.
```

3. 会议论文格式

```text
[序号] 著者. 题名. In文集名, 会议名, 会址, 开会时间. 编者, Eds., 出版地:出版者, 出版年: 页码范围.
```

4. 论文集中析出文献格式

```text
[序号] 作者. 题名[C]. //编者. 文集名. 出版地: 出版者, 出版年: 起-止页码.
```

5. 学位论文格式

```text
[序号] 作者. 题名[D]. 授予学位地: 授予学位单位, 出版年.
```

6. 电子文献格式（默认禁用）

```text
[序号] 作者. 题名[EB/OL]. (发表或更新日期). [引用日期]. 网址.
```

7. 专利格式

```text
[序号] 著者. 专利题名[P]. 专利号, 公告日期或公开日期.
```

正式会议论文和论文集文献首选第 4 条 `[C]. //` 格式。只有用户、学校模板或已有文档明确要求第 3 条 `In` 格式时，才使用第 3 条。不得按会议名、出版社名或论文集名写死规则；只要来源是正式、可查询、可核验的出版记录，并且条目满足对应模板，即可按相应类型处理。

作者规则：

- 列出前三名作者。
- 超过三名作者时列前三名后加 `et al`：前三名作者之间用半角逗号和空格分隔，第三名后加半角逗号、空格、`et al`，句末由条目末尾统一加英文句点；示例：`Vaswani A, Shazeer N, Parmar N, et al. Attention Is All You Need[C]. //...`
- 作者之间使用半角逗号。
- 中外作者均采用姓在前、名在后。
- 英文名使用姓在前、名缩写在后，如 `Vaswani A, Shazeer N, Parmar N, et al`。

## DOCX 机器审计规则

审计时使用只读工具，除非用户明确要求修复。

机器审计至少检查：

- 参考文献条目数量。
- 所有参考文献是否被正文引用。
- 是否存在越界引用、空引用、重复异常。
- 正文引用是否按首次出现顺序编号。
- 正文引用是否为上标、Times New Roman、小四/12 pt；若存在用户或模板覆盖，则单独说明。
- 每个引用点是否超过 3 条文献。
- 长串引用是否可以语义拆分。
- 参考文献表是否使用真实 Word 编号项。
- 正文引用是否优先为 Word“编号项/段落编号”交叉引用字段；若为书签 REF，则报告为 fallback；不得是普通文本或超链接锚点模拟。
- 审计 Word 编号项交叉引用时，忽略 Word 自动生成的 `_Ref数字` 隐藏书签；这类书签是 Word 交叉引用的正常实现细节，不得误报为书签 fallback。只有脚本 fallback 生成的自定义书签（如 `ref_<key>`）才报告为 fallback。
- 编号项交叉引用字段、显示文本和参考文献编号是否一致。
- 参考文献条目无首行缩进、无左缩进、悬挂缩进两个字符、双端对齐。
- 参考文献条目格式、标点、句末英文句点。
- 先识别参考文献类型，再按对应模板检查作者/著者、题名、类型标识、刊名或文集名、出版地、出版者、年份、卷期、页码、授予单位、专利号等通用字段。
- 是否存在默认禁用的电子文献痕迹，包括但不限于 `[EB/OL]`、URL、访问日期、预印本标识、网页型来源说明，或缺少正式出版地、出版者、页码等正式出版字段的电子页面条目。
- 写入前是否已有可查询、可核验的正式来源记录；不得只因为条目文本看似符合格式就通过。
- 编号定义 `w:numId` 是否存在并映射到完整编号定义。

不进行 DOCX 渲染，不生成页面 PNG、截图或其他页面图像，不执行视觉检查。审计结果以 OOXML、字段、编号、样式和文本抽取为准。

MCP server：

- `mcp-server/gb_reference_server.py`
  - 暴露 `search_formal_references`、`classify_gbt7714_candidate`、`filter_writable_candidates`。
  - 额外暴露 `search_and_verify_citation_points`、`build_manifest_draft_from_points`，支持“引用点 -> 候选 -> manifest 草稿”。
  - 复用 `scripts/检索正式文献.py` 的网络检索逻辑，不复制检索策略。
  - 默认代理 `http://127.0.0.1:7897`；MCP 未安装时不能作为 server 运行，但内部函数仍可被本地测试导入。
- `mcp-server/requirements.txt`
  - 仅列 MCP server 运行所需依赖。安装前仍需优先使用项目已有虚拟环境。

脚本：

1. `scripts/检索正式文献.py`
   - 只读网络检索工具，使用 Python 标准库调用 CrossRef、OpenAlex、PubMed，可显式加 `--source arxiv` 作为线索检索。
   - 默认通过 `http://127.0.0.1:7897` 代理访问网络；可用 `GBT7714_PROXY`、`HTTP_PROXY`、`HTTPS_PROXY` 或 `--proxy` 覆盖，`--proxy direct` 表示不使用代理。
   - 支持批量模式：`--citation-points-json` 读取引用点列表，自动生成查询词并输出每个引用点的可写入候选、拒收原因和部分成功清单。
   - 支持 `--allow-electronic`：用户或学校模板允许 `[EB/OL]` 时，电子线索可作为 `[EB/OL]` 候选进入草稿；默认不写入，但仍保留在检索结果中给模型查阅。
   - 输出 JSON 候选清单，含 `suggested_gbt_type`、`formal_record`、DOI/PMID/URL、来源说明和拒写提示。
   - `formal_record=false` 或 `suggested_gbt_type=CLUE_ONLY` 的候选不得直接写入；必须继续核验 DOI、出版方、会议/期刊、学位论文、专利或权威数据库记录。
2. `scripts/gbt.py`
   - 英文 ASCII 调度器，适合其他电脑、cmd、远程终端、非 UTF-8 shell 或不方便输入中文文件名的环境。
   - 命令映射：`search`、`audit`、`extract`、`apply`、`overview`、`reorder`、`split`。
   - 示例：`python -B scripts/gbt.py search "Monte Carlo Tree Search" --source crossref`。
3. `scripts/抽取文档引用.py`
   - 只读。
   - 输出正文引用点、上下文、参考文献编号项、交叉引用字段、字段目标类型、字体、字号和上标状态。
4. `scripts/审计参考文献.py`
   - 只读。
   - 检查编号项、编号项交叉引用、引用覆盖、长串引用、字体字号、上标、参考文献格式、越界引用和未引用条目。
   - 必须把"用户/模板指定字体导致不同于默认 Times New Roman 小四"与"格式错误"分开报告。
5. `scripts/生成参考文献概览.py`
   - 维护 `参考文献/参考文献概览.md` 和 `参考文献/任务状态.json`。
   - 支持从任意阶段恢复任务。
   - 写入 `正文字体`、`正文字号_中文` 和 `正文字号_磅值`，避免后续任务丢失字体偏好。
6. `scripts/应用参考文献.py`
   - 根据结构化 JSON/YAML 重建或修复 DOCX 正文引用、编号项交叉引用、真实参考文献编号和参考文献节格式。
   - 可使用占位 DOCX、临时 JSON 或 OOXML 副本作为内部处理中间件，但这些文件不是交付件；最终回复只交付最终 DOCX 和必要报告，除非用户明确要求查看中间文件。
   - 必须支持局部修改：只改指定章节、指定段落、指定引用组或指定参考文献条目。
   - 写入前必须校验每条 manifest 记录含有 DOI、出版社页面、会议/期刊官网、数据库记录、权威索引记录或其他可查询核验来源；无核验来源时不得写入。
   - 修改含正文引用的段落时不得整段清空重建；只能替换普通文本运行，必须保留上标运行、Word 字段、书签和交叉引用结构，并在写入后确认引用编号、上标状态、字段和书签数量未被破坏。
   - 默认把正文引用设置为 Times New Roman 小四、上标。
   - 参考文献条目段落无首行缩进、无左缩进、悬挂缩进两个字符、双端对齐。
   - 默认 `--method auto`：先尝试 Word 编号项交叉引用，失败后自动使用书签 REF fallback；`--method numbered-item` 可强制失败不降级，`--method bookmark` 可直接使用 fallback。
   - Word 编号项交叉引用依赖 Windows + Microsoft Word + PowerShell/COM；macOS/Linux 或无 Word 环境下，`--method auto` 会降级到书签 REF，`--method numbered-item` 会失败并报告原因。
7. `scripts/重排序号.py`
   - 根据正文首次出现顺序重排参考文献表和正文交叉引用字段。
   - 删除未引用项或补齐引用后都可使用。
8. `scripts/拆分引用组.py`
   - 找出超过阈值的引用组。
   - 根据模型名、方法名、工具名、名词短语和文献标题关键词提出拆分建议；优先给出短语级落点，避免默认建议放在子句或句末。
   - 默认不自动改写事实，只移动或拆分引用字段。
